# 세금 관리 앱뷰 프로토타입 구현 계획

작성일: 2026-05-28  
상태: Draft  
범위: 기능 없는 시나리오 기반 세금 관리 웹앱 프로토타입. 앱뷰만 구현.

## 1. 조사 요약

### 핵심 레퍼런스 인사이트
- Toss Payments는 화면 하나에 명확한 목표 하나를 두는 제품 원칙을 퍼널 설계에 연결한다. 이 프로토타입도 “시나리오 선택”과 “시나리오 상세 읽기”를 분리한다.
- Toss Simplicity 제작기는 모바일 사용 비중을 근거로 모바일 중심 설계와 짧고 명확한 콘텐츠 구성을 강조한다. 앱뷰 전용 요구와 잘 맞으므로 모바일 퍼스트로 간다.
- Toss 시니어 UX 리서치는 질문형 문구, 숨겨진 스크롤, 애매한 진입점이 오인지로 이어질 수 있음을 보여준다. 세금 정보 화면은 행동형 문구, 명확한 구조, 스크롤 가능성 인지를 우선한다.
- Naver Pay 디자인 가이드는 브랜드 요소를 매체 특성에 맞게 적용하되 가독성을 고려하는 방향을 보여준다. 본 앱은 네이버페이 색을 복제하지 않고, 흰 배경과 블루 포인트로 금융 신뢰감을 만든다.
- 한국 UX/UI 블로그와 요즘IT 계열 글들은 사용자-비즈니스-맥락, 컴포넌트/토큰 기반 일관성, 와이어프레임과 프로토타입의 분리를 공통적으로 강조한다.

## 2. 요구사항 요약

- 데이터: 두 개 시나리오 기반의 목킹 텍스트만 표시
- 비즈니스 로직: 구현하지 않음
- 화면: 웹뷰/데스크톱 화면 구현 금지, 앱뷰만 구현
- 요소 제한: 사용자 요구사항 또는 시나리오 콘텐츠에 없는 화면 요소 추가 금지
- UX: 선택/전환/피드백을 위한 상호작용과 애니메이션 포함
- API: 추후 백엔드 연동을 가정한 mock API 명세 작성
- 우선순위: 로직 정합성보다 디자인 완성도
- 레퍼런스 방향: Toss, Naver Pay처럼 군더더기를 줄인 금융/결제 앱 UI

## 3. 제품/UX 방향

### 디자인 원칙
1. One thing per screen: 한 화면에 하나의 목적만 둔다.
2. No extra elements: 요구사항 밖의 메뉴, 통계, 로그인, 알림, 광고, 장식 아이콘을 넣지 않는다.
3. Mobile-first app view: 320~430px 폭에서 가장 완성도 있게 보이도록 설계한다.
4. Calm finance UI: 흰 배경, 블루 포인트, 넉넉한 여백, 높은 대비로 신뢰감을 준다.
5. Meaningful motion: 선택/전환/확인감을 주는 최소 모션만 사용한다.

### 사용자 흐름
1. 사용자는 앱뷰 홈에서 두 개 시나리오를 본다.
2. 한 시나리오를 선택한다.
3. 선택된 시나리오 상세 목킹 텍스트가 앱 전환처럼 자연스럽게 나타난다.
4. 사용자는 다른 시나리오로 전환하거나 이전 선택 화면으로 돌아간다.

## 4. 화면 계획

### Screen A: 시나리오 선택
- 포함 요소:
  - 앱/화면 제목
  - 시나리오 항목 2개
  - 각 시나리오의 짧은 목킹 요약 텍스트
- 금지 요소:
  - 로그인, 프로필, 알림, 검색, 하단 탭, 실제 계산 결과, 추천 카드
- 인터랙션:
  - 항목 press feedback
  - 선택 시 상세 화면으로 slide/fade 전환

### Screen B: 시나리오 상세
- 포함 요소:
  - 선택한 시나리오 제목
  - 선택한 시나리오 목킹 본문
  - 요구사항상 필요한 경우에만 뒤로/다른 시나리오 보기 컨트롤
- 인터랙션:
  - 본문 등장 모션
  - 뒤로 이동 시 반대 방향 전환

## 5. Mock API 명세 초안

실제 서버는 만들지 않지만, 프론트엔드는 백엔드 응답을 받는 구조처럼 분리한다.

### `GET /api/scenarios`
목적: 시나리오 선택 화면에 표시할 두 개 시나리오 목록을 반환한다.

응답:
```json
{
  "scenarios": [
    {
      "id": "scenario-a",
      "title": "시나리오 A 제목",
      "summary": "시나리오 A 요약 목킹 텍스트"
    },
    {
      "id": "scenario-b",
      "title": "시나리오 B 제목",
      "summary": "시나리오 B 요약 목킹 텍스트"
    }
  ]
}
```

### `GET /api/scenarios/{scenarioId}`
목적: 선택된 시나리오 상세 목킹 텍스트를 반환한다.

응답:
```json
{
  "id": "scenario-a",
  "title": "시나리오 A 제목",
  "body": [
    "상세 목킹 텍스트 문단 1",
    "상세 목킹 텍스트 문단 2"
  ]
}
```

### API 제약
- `scenarios.length`는 2로 고정
- 계산 필드, 금액 산출 필드, 실제 세금 판단 필드는 만들지 않음
- 실패 응답은 구현 필수가 아니며, 테스트용 fixture로만 둘 수 있음

## 6. 구현 단계

1. 신규 프론트엔드 스캐폴드 결정 및 생성
   - 앱 소스가 없으므로 React/Vite 또는 Next.js 중 가벼운 선택
   - 앱뷰 폭을 제한하는 루트 레이아웃 작성

2. 디자인 토큰 정의
   - 색상: white background, blue accent, text, muted border
   - spacing: 8px scale, 16px side padding
   - radius/elevation/motion duration/easing

3. Mock API/fixture 작성
   - `scenarios` 목록 fixture
   - `scenarioDetail` fixture
   - API 계약 문서와 타입을 일치시킴

4. 앱뷰 화면 구현
   - `ScenarioSelector`
   - `ScenarioDetail`
   - 상태는 선택된 시나리오 ID 정도로 제한

5. 인터랙션/모션 구현
   - press state
   - screen transition
   - body reveal
   - `prefers-reduced-motion` 대응

6. 화면 요소 제한 검수
   - 요구사항 외 UI가 들어갔는지 체크리스트로 검토
   - 장식/편의 요소가 추가되면 제거

7. 검증
   - 앱뷰 크기 스모크 확인
   - 두 시나리오 모두 선택/상세 전환 확인
   - 목킹 API 타입/문서 일치 확인
   - 접근성 기본 확인: 버튼 의미, 포커스, 대비, reduced motion

## 7. 수용 기준

- 앱뷰에서만 완성도 있게 보이고, 데스크톱 전용 레이아웃을 만들지 않는다.
- 시나리오 2개 외 데이터가 없다.
- 실제 세금 계산/저장/인증/결제 로직이 없다.
- 화면 요소는 제목, 시나리오 항목, 시나리오 텍스트, 요구된 이동/선택 컨트롤로만 구성된다.
- 상호작용은 사용자가 선택과 전환을 인지할 수 있을 만큼 충분히 부드럽다.
- Mock API 명세와 fixture/type이 일치한다.

## 8. 리스크와 대응

- 리스크: “요소 추가 금지”와 “UX 개선”이 충돌할 수 있음
  - 대응: 모션과 상태 피드백은 기존 요구 요소에만 적용하고 새 UI 장식은 만들지 않음
- 리스크: 세금 앱 맥락 때문에 실제 기능처럼 보일 수 있음
  - 대응: 목킹 텍스트임을 콘텐츠 수준에서 명확히 하되, 별도 배너 같은 추가 요소는 요구 시에만 추가
- 리스크: 시나리오 원문이 길면 앱뷰 가독성이 떨어짐
  - 대응: 문단 단위 spacing과 제목 위계로 처리하고, 임의 요약/추가 정보는 만들지 않음

## 9. 구현 전 필요한 사용자 제공값

- 시나리오 A/B의 제목, 요약, 상세 목킹 텍스트
- 선호하는 기술 스택이 있다면 지정. 없으면 가벼운 React/Vite 기준으로 진행 가능.
