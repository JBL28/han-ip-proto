# 공제 상세보기 페이지 구현 계획

## 0. UI 기준

- 공제 상세보기 페이지는 **메인화면과 절세추천 화면을 레퍼런스**로 삼는다.
- 상세 페이지라도 독립 디자인을 만들지 않고 AppShell 표준 비율, 카드 radius, gap, 폰트/색상 토큰을 유지한다.
- 법령/조건 정보가 길어질 수 있으므로 카드 단위로 정보 위계를 나누되, 사용자가 한 화면에서 핵심 금액과 다음 행동을 먼저 이해하게 한다.
- 큰 카드/섹션은 `AppCard` 또는 동일한 카드 토큰을 사용한다.
- 그라데이션은 금지한다. 정보 강조는 단색 배지, 굵기, 여백으로 처리한다.

## 3. 데이터 원칙

- 공제 상세 내용은 페르소나별 추천 공제 데이터와 연결한다.
- `mockdatas/index.md`에 있는 화면용 요약 문구를 UI 본문 기준으로 삼는다.
- 법령 텍스트 변환본은 `EvidenceList` 또는 source 주석에 연결하고, 긴 원문을 임의 요약해 새로운 조건처럼 쓰지 않는다.
- PDF fallback 추출이 빈약한 파일은 `mockdatas/index.md`를 canonical source로 사용한다.
- 실제 대상 여부 판정/세액 계산/신청 제출은 구현하지 않는다.

## 4. 이미지 에셋 생성 계획

필요한 투명 배경 PNG:

- 공제 상세 상단 일러스트: 문서+체크 표시
- 월세/주거 공제용 집+영수증
- 교육비 공제용 책/연필
- 의료비 공제용 안경/의료 영수증
- 장려금/지원금용 동전+체크

저장 위치:

- `src/assets/deduction-detail/`

파일명:

- `deduction-document.png`
- `rent-receipt.png`
- `education-receipt.png`
- `medical-receipt.png`
- `benefit-coins.png`

생성 프롬프트 공통 규칙:

- 투명 배경 PNG, 텍스트/숫자 없는 오브젝트 일러스트.
- 흰색, 연파랑, 네이비, 작은 노란 포인트 정도만 사용.
- 실제 국세청/정부 로고를 넣지 않는다.
- 이미지에 카드 프레임이나 그림자를 넣지 않고 CSS에서 처리한다.

## 5. 컴포넌트 구조

`DeductionDetailPage.jsx`

- `useParams()`로 `personaId` 확인.
- 현재 라우트에는 상세 id param이 없으므로, 1차 구현은 페르소나별 대표 공제 1개를 보여준다.
  - 한버팀 기본: `근로장려금 및 자녀장려금` 또는 와이어프레임 공제명
  - 김갓생 기본: `월세액 세액공제` 또는 와이어프레임 공제명
- 향후 상세 id가 필요하면 `/:personaId/deduction-detail/:deductionId`로 확장한다.

`DetailHeroCard`

- 공제명, 핵심 혜택, 짧은 설명, 이미지.
- hero 내부 CTA는 와이어프레임에 있을 때만 둔다.

`DetailInfoSection`

- `대상 조건`, `필요 서류`, `신청 방법`, `근거`를 카드/accordion 형태로 표시.
- accordion을 쓰면 open/close UI 상태는 mock으로 동작해야 한다.

`EvidenceList`

- source 파일명과 조항명을 짧게 표시한다.
- 원문 긴 문장은 상세 접힘 영역에만 둔다.

## 6. 레이아웃 계획

- 상단 hero card → 조건/서류/근거 section → 하단 CTA 순서.
- CTA가 sticky이면 bottom nav와 겹치지 않게 한다.
- 긴 법령명은 줄바꿈 가능하게 하고 카드 밖으로 넘치지 않게 한다.
- 섹션 간 외부 여백은 부모 gap만 사용한다.
- `AppCard` 내부 padding은 공통 규칙을 따르고 개별 카드 root margin을 만들지 않는다.

## 7. 링크 동작

- 뒤로가기: 브라우저 history 또는 `/${personaId}/tax-saving`
- `챗봇에게 물어보기`: `/${personaId}/tax-chat`
- 정책/지원금 연결이 필요한 CTA: `/${personaId}/policies`
- 신청/확인 CTA는 실제 신청 로직 없이 mock 상태 또는 알림 페이지 이동으로 제한한다.

## 8. 검증 계획

- `npm run validate`
- 그라데이션 금지 검색:
  - `grep -R "linear-gradient\|radial-gradient\|conic-gradient" -n src dist/index.html dist/assets/*.css dist/assets/*.js || true`
- 공제명/금액/법령명이 `mockdatas/index.md`와 충돌하지 않는지 확인
- 긴 법령/설명 텍스트가 320px에서 넘치지 않는지 확인
- CTA 라우팅 동작 확인
- 와이어프레임 텍스트 누락 수동 대조
